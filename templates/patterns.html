Pattern · 01

监督者模式 Supervisor

一个中心化的大模型"调度员"掌控全局：它读取全部上下文，决定每一步派哪个专家出场，并判断何时收工。这是 LangGraph 最经典、也是本项目聊天模式采用的模式。

🧭

Supervisor 监督者

中心化 · LLM 路由

中央监督者是一个 LLM 节点，每轮读取全部历史，用提示词自行判断"该派谁、是否 FINISH"。专家是被调度的叶子节点，完成后回到监督者形成循环。

拓扑 
        ┌─────────────┐
用户 ──▶│  Supervisor │◀──────────┐
        │   (LLM)     │           │
        └──────┬──────┘           │
      派发任务 │                  │ 回到监督者
   ┌──────────┼──────────┐       │ 判断 FINISH
   ▼          ▼          ▼       │
┌──────┐  ┌──────┐  ┌──────┐    │
│Coder │  │ Res. │  │Writer│────┘
└──────┘  └──────┘  └──────┘
   条件边: supervisor → {coder|researcher|writer|FINISH}

● 优势

  • 控制力强，全局可见
  • 易于调试与追踪路由
  • 终止逻辑集中可控

● 劣势

  • 每轮多一次 LLM 调用
  • 监督者是性能瓶颈
  • 专家过多时路由失准
🏢

Hierarchical 层级

嵌套监督者

监督者的监督者。顶层监督者把大任务拆给"团队监督者"，每个团队监督者再调度自己的专家子图。本质是 Supervisor 的递归组合，适合规模庞大的组织化分工。

拓扑 
          ┌──────────────┐
   用户 ──▶│  Top Sup.    │  拆解大任务
          └──┬────────┬──┘
        ┌────┘        └────┐
        ▼                  ▼
  ┌──────────┐       ┌──────────┐
  │研发团队   │       │内容团队   │
  │Supervisor│       │Supervisor│
  └─┬───┬─┬──┘       └──┬───┬───┘
    ▼   ▼ ▼              ▼   ▼
  Coder QA ...       Writer Editor
   "监督者的监督者" = 多层 StateGraph 嵌套

● 优势

  • 可无限扩展规模
  • 职责边界清晰分层
  • 子图可独立复用

● 劣势

  • 调用链长、延迟叠加
  • 上下文跨层传递复杂
  • 过度设计风险高
🐝

Swarm 群体

去中心化 · Handoff

没有中央调度员。每个专家自主判断"这事我搞不定，交给 X"，通过 handoff 把控制权直接移交给最合适的同行。路由发生在工具调用层，而非额外的 LLM 节点。

拓扑 
  无中心节点，专家间直接 handoff：

   ┌────────┐  handoff   ┌─────────┐
   │ Coder  │───────────▶│Researcher│
   └────┬───┘            └────┬─────┘
        │     handoff         │
        ▼                     ▼
   ┌────────┐  handoff   ┌─────────┐
   │ Writer │◀───────────│  ...    │
   └────────┘            └─────────┘
   每个 agent 调用 transfer_to_X() 工具移交控制

● 优势

  • 无中心瓶颈
  • 专家多时开销更低
  • 对等协作更自然

● 劣势

  • 路由不直观、难调试
  • 可能死循环移交
  • 缺乏全局视角
🕸️

Network 网状

全互联 · 自组织

最自由的拓扑：任意 agent 都可与其他任意 agent 直接通信，没有固定调度顺序。靠共享状态和动态条件边自组织，常用于复杂的协作型工作流和模拟社会。

拓扑 
  全互联，任意点对任意点：

      ┌────────┐
   ┌─▶│ Agent A│◀┐
   │  └───┬────┘ │
   │      │      │
   │  ┌───▼────┐ │
   └──│ Agent B│─┘
      └───┬────┘
          │
   ┌──────▼───┐
   │ Agent C  │──(可连回 A/B)
   └──────────┘
   动态条件边，由状态/LLM 决定下一步走向

● 优势

  • 表达力最强
  • 贴近真实协作
  • 支持并行涌现

● 劣势

  • 极易失控/死循环
  • 行为难预测难复现
  • 调试与成本最高

Pattern · 03

该选哪一种？

没有银弹。根据你的核心诉求，从下面几个判断题出发快速锁定。

选型决策树

回答这些问题，匹配度最高的模式就是你的起点。

Q1.需要全局可见、可控可调试的路由？
是 → Supervisor。单点调度，每一步都可追踪，最适合需要确定性流程的工作流。
Q2.专家数量多到需要分组管理？
是 → Hierarchical。把专家按职能聚成子团队，每个团队一个监督者，顶层再统筹。
Q3.专家是对等的、想让它们自己互相移交？
是 → Swarm。省掉中心节点，专家通过 handoff 工具直接转交，省时省 token。
Q4.需要任意角色之间都能自由协作（如模拟辩论/社会）？
是 → Network。表达力最强，但要自己兜底死循环和成本。
Q5.对延迟和 token 成本敏感？
优先 Swarm > Supervisor，避免 Hierarchical 的多层调用叠加。
Q6.不确定 / 想先跑通？
从 Supervisor 起步——它最直观、最可控，后续随时可演化为其他模式。
Pattern · 04

本项目用了哪种？

这个 LangGraph Demo 同时实现了两个图，它们分别对应不同模式的"变体"。

💬 聊天模式（graph.py）

监督者每轮读全部历史，用确定性代码判定 FINISH（防止推理模型乱结束），派发给 coder / researcher / writer 三专家之一，回答后回到监督者。

✓ 标准 Supervisor 模式（带确定性兜底）

🎯 圆桌讨论（discussion.py）

协调者每轮挑一个按主题动态招募的专家发言，固定 N 轮后进入总结者。形态像 Supervisor，但终止靠轮次计数器，不靠 LLM 自主判断。

✓ Supervisor 形态 + 固定循环 + 独立总结节点

💡 关键洞察：同一个项目里，"谁决定何时结束"是区分模式可靠性的核心。聊天模式把 FINISH 交给代码兜底；讨论模式干脆把结束权完全交给轮次计数器——把不可靠的 LLM 自主判断从控制流里拿走，是工程上保证多智能体系统不"跑飞"的常用手段。
